# Scene 与本模块兼容性说明

## 冲突分析

### 1. 功能重叠
- **Scene**: 通过修改 CPU 调度器参数、频率策略来优化性能
- **本模块**: 通过线程亲和性绑定 CPU 核心来优化性能
- **结论**: 两者作用层面不同，理论上可以共存

### 2. 可能的冲突点

#### CPU核心控制
```
Scene: 可能限制CPU核心数量（省电模式）
本模块: 需要所有核心可用
```

#### 调度器参数
```
Scene: 修改 walt 调度器参数
本模块: 依赖 walt 调度器的默认行为
```

#### 性能模式
```
Scene: 提供多种性能配置
本模块: 固定的线程绑定策略
```

## 兼容方案

### 方案1：Scene + 本模块（推荐）

**优势**: 结合两者优点
- Scene 负责 CPU 频率和调度器
- 本模块负责线程亲和性

**配置建议**:

1. **Scene 设置**
   ```
   模式: 性能模式 或 均衡模式
   CPU策略: 不要限制核心数量
   调度器: 保持 walt
   ```

2. **本模块设置**
   ```
   保持默认配置即可
   模块会自动延迟启动以兼容Scene
   ```

### 方案2：仅使用本模块

如果 Scene 导致冲突，可以：
```bash
# 在 Magisk 中禁用 Scene
# 或卸载 Scene
```

### 方案3：仅使用 Scene

如果本模块效果不佳，可以：
```bash
# 在 Magisk 中禁用本模块
```

## 推荐配置（Scene + 本模块）

### Scene 配置

#### 性能模式（游戏时）
```
CPU调度: 性能
GPU调度: 性能
核心数量: 全开（8核）
温控: 适中
```

#### 均衡模式（日常）
```
CPU调度: 均衡
GPU调度: 均衡
核心数量: 全开（8核）
温控: 标准
```

#### 省电模式（待机）
```
CPU调度: 省电
GPU调度: 省电
核心数量: 建议全开（让本模块控制）
温控: 激进
```

### 本模块配置

保持默认即可，模块会：
1. 检测 Scene 是否安装
2. 延迟15秒启动，让 Scene 先初始化
3. 在 Scene 基础上应用线程绑定

## 冲突排查

### 如果出现卡顿

1. **检查 Scene 配置**
   ```bash
   # 确保没有限制核心数量
   cat /sys/devices/system/cpu/cpu*/online
   # 应该全部显示 1
   ```

2. **检查线程绑定**
   ```bash
   # 查看游戏进程
   ps -A | grep 游戏包名
   
   # 查看线程绑定（替换PID）
   cat /proc/PID/status | grep Cpus_allowed_list
   ```

3. **临时禁用测试**
   ```bash
   # 禁用本模块
   killall AppOpt
   
   # 或禁用 Scene
   # 在 Magisk 中操作
   ```

### 如果发热严重

1. **调整 Scene 温控**
   ```
   Scene -> 温控策略 -> 激进
   ```

2. **调整本模块配置**
   ```bash
   # 编辑 applist.conf
   # 减少超大核(7)的使用
   # 例如：
   # 游戏{UnityMain}=5-6  # 原来是6-7
   ```

### 如果续航变差

1. **Scene 使用省电模式**
2. **本模块保持默认**（已针对省电优化）

## 常见问题

**Q: Scene 和本模块哪个优先级高？**
A: Scene 先启动（调度器层面），本模块后启动（线程层面），互不冲突。

**Q: 可以只用其中一个吗？**
A: 可以，但结合使用效果更好。

**Q: 如何判断是否冲突？**
A: 查看日志，测试游戏性能，对比开启前后差异。

**Q: 冲突了怎么办？**
A: 先禁用本模块测试，如果 Scene 够用就不需要本模块。

**Q: 两个都用会不会更耗电？**
A: 不会，合理配置反而更省电。
